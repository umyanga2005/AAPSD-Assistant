# Complete System Working Report: AAPSD-Assistant

## 1. Overview

The AAPSD-Assistant is an agentic DevOps platform designed to consolidate infrastructure monitoring, continuous integration pipelines, incident management, and AI-driven troubleshooting into a single, cohesive dashboard.

The system is split into two primary applications residing within a monorepo structure:

- **Frontend App (`@aapsd/web`)**: A responsive React SPA built with Vite, utilizing Tailwind CSS for modern aesthetics and Firebase Web SDK for secure authentication.
- **Backend Service (`@aapsd/api`)**: A Fastify-based Node.js backend using Drizzle ORM to interface with a Supabase PostgreSQL instance, providing REST endpoints and real-time WebSocket capabilities.
- **Packages**: Shared packages such as `@aapsd/diagnosis` encapsulate integrations with external systems (e.g., Kubernetes, Prometheus, GitHub Actions).

## 2. Authentication Flow

The system recently migrated from mock local authentication to a secure Firebase/Supabase architecture:

- **Frontend**: Utilizes `firebase/auth` for managing user sessions. A custom, Tailwind-styled `Login.tsx` handles email/password and Google OAuth providers. The global `App.tsx` router listens to `onAuthStateChanged` to securely gate application access.
- **Backend Middleware**: The Fastify backend relies on the `firebase-admin` SDK. The `devAuthPreHandler` intercepts incoming requests, verifying the Firebase JWT sent via the `Authorization` header (`Bearer <token>`). Validated tokens are synced against the local PostgreSQL `users` table to maintain internal consistency and role-based access control.

## 3. Database Architecture (Supabase PostgreSQL)

All schema changes are managed via `drizzle-orm` and `drizzle-kit`. Key entities include:

- **`users`**: Developer records synchronized via Firebase UIDs/Emails.
- **`projects` / `project_members`**: Tenancy scoping mechanisms.
- **`user_integrations`**: Securely stores third-party OAuth tokens (e.g., GitHub Access Tokens).
- **`requests` / `audit_events`**: Detailed logging for AI diagnosis requests and infrastructure state changes.

_Note on Network Architecture: Windows development environments experience DNS issues connecting to default Supabase IPv6 endpoints; therefore, the system connects via an IPv4 Transaction Pooler._

## 4. Third-Party Integrations

- **GitHub Actions**: Controlled by the `RealGitHubAdapter`. Users authenticate their GitHub accounts on the frontend `Settings.tsx` page. The OAuth callback (`/api/auth/github/callback`) securely stores the user's `access_token` in the database, allowing dynamic instantiation of API clients per-request.
- **Kubernetes / Prometheus**: Handled via their respective adapters in `@aapsd/diagnosis`, pulling live node telemetry, deployment health, and resource thresholds.

## 5. UI/UX and Theming

The frontend adheres to strict modern design standards:

- **Vibrant & Dynamic**: Deep "brand-dark" backgrounds paired with subtle glow effects, glassmorphism (`glass-panel`), and micro-animations (`animate-fade-in`).
- **Responsive Layouts**: A collapsible side navigation bar scales seamlessly across desktop and mobile devices.

## 6. Testing & CI

The codebase employs Vitest for robust unit testing (`npm run test`). Test environments use mocked SDK boundaries (`setup.ts`) to ensure deterministic executions free from real API side-effects. The application passes all build phases without type errors.
