# Manual Testing Guide: AAPSD Assistant

This guide provides a complete, step-by-step walkthrough for manually testing the AAPSD Assistant on your local machine, now that real Supabase Authentication and AI integrations are fully implemented.

## 1. Prerequisites

Before starting, ensure your Supabase database is properly linked and the API keys are correct.
1. Open `apps/api/.env` and `apps/web/.env`.
2. Verify that `SUPABASE_URL` and the respective keys (`SUPABASE_SERVICE_ROLE_KEY` for API and `VITE_SUPABASE_ANON_KEY` for web) are correctly formatted without any hidden spaces.
3. Ensure your local LLM credentials (`GROQ_API_KEY`) are present in `apps/api/.env`.

## 2. Starting the System

You need to run both the Backend API and the Frontend Web application simultaneously. Open two separate PowerShell terminal windows.

**Terminal 1 (Backend API):**
```bash
npm run dev --workspace=@aapsd/api
```
*Wait until you see `Server listening at http://127.0.0.1:3001`.*

**Terminal 2 (Frontend Web):**
```bash
npm run dev --workspace=@aapsd/web
```
*The UI will be served at `http://localhost:5173`.*

## 3. Testing Authentication (Supabase)

1. Open your browser and navigate to `http://localhost:5173`.
2. You should be automatically redirected to the **Login Page**.
3. **Sign Up / Log In**: 
   - Since this is your first time using the real Supabase backend, you need an account. 
   - *Note: If you haven't enabled email confirmations in your Supabase project settings, you can log in immediately after signing up. If email confirmations are enabled, check your email or disable them in the Supabase Dashboard (Authentication -> Providers -> Email).*
4. Once logged in, you will be redirected to the main Dashboard.

## 4. Testing the Dashboard & Infrastructure

The system attempts to fetch real data using the backend adapters.
- **Infrastructure Page**: Navigate to the "Infrastructure" tab. It will attempt to connect to your local Kubernetes cluster using your `.kube/config`. If you have a local cluster (like Minikube or Docker Desktop) running, you should see your Deployments and Pods listed.
- **Pipelines Page**: Navigate to the "Pipelines" tab. It will connect to GitHub using the `GITHUB_TOKEN` defined in your `.env` and fetch recent workflow runs.

## 5. Testing the AI Investigator (Groq LLaMA 3.1)

1. Navigate to the **Incidents Page** from the sidebar.
2. You will see a list of detected alerts (e.g., `Cluster Node NotReady`).
3. Click the **"Investigate"** button next to one of the critical alerts.
4. **Expected Behavior**:
   - The UI will open the right-side Assistant panel.
   - A request securely authenticated with your Supabase Token is sent to the backend.
   - The backend `@aapsd/diagnosis` engine queries Groq's LLaMA 3.1 model.
   - The AI will return a root cause analysis and a suggested runbook/fix.
   - *Note: If the AI returns "Insufficient Evidence", it means your local environment didn't have enough real logs for that specific mock alert.*

## 6. Testing the Audit Logs

To verify that the system complies with the security requirements of your Methodology:
1. Navigate to the **Audit Logs Page**.
2. **Expected Behavior**: You should see a chronological list of actions you just performed.
   - You should see an event for when you logged in.
   - You should see a `diagnosis.requested` event from when you clicked the "Investigate" button.
   - This proves the backend `devAuthPreHandler` successfully identified your Supabase User ID and recorded your actions to the PostgreSQL database via Drizzle ORM.

---

> [!TIP]
> **Troubleshooting Real Integrations**
> If you don't see any GitHub pipelines or Kubernetes data, ensure your `GITHUB_TOKEN` and local K8s cluster are properly configured. Currently, the system falls back to your `.env` variables. In the next phase, we will build a UI for users to input their own personal integration tokens directly into the `user_integrations` table!
