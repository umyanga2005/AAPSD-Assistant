# Bug Fixes & Resolution Report

This report outlines all the technical issues and bugs we encountered during our session and exactly how they were resolved to get the Smart DevOps Assistant fully operational.

### 1. Groq API Integration & Security Warning
**Bug:** The backend was configured to use OpenRouter, and it had an insecure `NODE_TLS_REJECT_UNAUTHORIZED=0` flag left in the `.env` file.
**Fix:** We updated the `OPENAI_BASE_URL` to point to `https://api.groq.com/openai/v1`, keeping the existing OpenAI-compatible request shape intact. We also securely removed the `NODE_TLS_REJECT_UNAUTHORIZED=0` flag from the environment.

### 2. GitHub Actions CI/CD Billing Lock
**Bug:** The GitHub Actions pipeline failed to run with the error: `The job was not started because your account is locked due to a billing issue.`
**Fix:** We bypassed GitHub's cloud compute restrictions by setting up a **Self-Hosted Runner** directly on your Windows PC. We changed the workflow file to use `runs-on: self-hosted`, allowing the pipeline to execute locally and entirely free of charge.

### 3. Missing Root Test Script
**Bug:** When the local runner executed `npm test`, it failed with `Missing script: "test"`.
**Fix:** We added `"test": "npm run test --workspaces --if-present"` to the root `package.json`, allowing the pipeline to correctly delegate tests to the individual `api`, `web`, and `diagnosis` workspaces.

### 4. Vitest & Undici Compatibility Crash
**Bug:** The API tests crashed spectacularly with `webidl.util.markAsUncloneable is not a function`. This was an incompatibility between the latest version of the `undici` networking library (v7/v8) and the `vitest` testing framework.
**Fix:** We explicitly downgraded the `undici` dependency in the `@aapsd/diagnosis` package to stable version `6.19.2`, completely eliminating the uncloneable crash.

### 5. Kubernetes Local Connection Failure (`fetch failed`)
**Bug:** The API server was unable to fetch data from your local Kubernetes cluster, throwing a `fetch failed` error due to Kubernetes using a self-signed certificate, which Node's native `fetch` rejects by default.
**Fix:** We imported `fetch` and `Agent` directly from `undici` in `packages/diagnosis/src/k8s-adapter/real.ts`. We created a custom networking agent with `rejectUnauthorized: false` and passed it into the `undici.fetch` call, successfully bridging the connection to the Kubernetes API.

### 6. Infrastructure & Dashboard UI Data Mapping
**Bug:** After fixing the Kubernetes connection, the UI displayed `0`s, `NaN`s, and "Unknown" for pod statuses. The UI expected custom fields like `desired` and `restarts`, but the Kubernetes adapter returns raw K8s API fields like `replicas` and nested `containers` arrays.
**Fix:** We rewrote the UI rendering logic in `InfrastructurePage.tsx` and `Dashboard.tsx` to correctly traverse the Kubernetes API schema. It now reads `replicas`, calculates `availableReplicas >= replicas`, reads pod `phase` (Running/Pending), and reduces the `restartCount` from the nested container statuses.

### 7. UI Unit Test Failure
**Bug:** The frontend CI pipeline failed because `InfrastructurePage.test.tsx` was still using mock data modeled after the old, broken data schema, so it couldn't find the expected 'CrashLoopBackOff' text on the screen.
**Fix:** We updated the mock data inside the test file to perfectly mimic the real Kubernetes API schema, resulting in a 100% green test suite.

### 8. Unresponsive "Investigate" Button
**Bug:** The "Investigate" button on the Incidents page was purely cosmetic and didn't trigger any action.
**Fix:** We added an `onClick` handler that dynamically routes the user to the Assistant screen (`/assistant`) and passes the incident title via a URL query parameter (`?incident=...`). We also updated the `AssistantScreen` to read this parameter and automatically pre-fill the chat input so the AI can begin diagnosing immediately.
